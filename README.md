# 開成祭チケット発行システム 開発・保守者向け技術仕様書 (README)

本システムは、生徒が入力した Microsoft Forms の回答データと、チケット管理用の固有ID・PINコードをローカル環境で名寄せ・紐付け・データ結合し、Web管理システムへのインポート用データおよび通知書印刷用（Word差し込み印刷用）データを生成するためのクライアントサイド・アプリケーションです。

---

## 1. システム概要とアーキテクチャ

本システムは、セキュリティ制限の厳しい公務用PC等での動作を想定し、**シングルHTMLファイル（HTML/CSS/JavaScript）**で実装されています。

*   **動作環境**: モダンブラウザ（Google Chrome, Microsoft Edge等）
*   **外部依存関係**: なし（ライブラリを使用せず、Vanilla JSにて記述）
*   **通信処理**: サーバーサイド通信は一切行わず、すべての処理はブラウザのメモリ上で完結します（完全ローカル動作）。
*   **データ永続化**: 生成済みのID/PINや生成履歴、重複チェック用データはブラウザの `LocalStorage` を使用して保持します。

---

## 2. 主要モジュールとアルゴリズムの解説

### 2.1 表記揺れ吸収と名寄せ (標準化ロジック)
生徒によって入力形式が異なる「年組番号」をキーとして名寄せを行うため、以下のクレンジング処理を実装しています。

*   **関数**: `standardizeClassNum(str)`
*   **処理手順**:
    1.  全角英数字を半角英数字へ置換（例：`１` → `1`, `Ａ` → `A`）。
    2.  漢数字を半角算用数字へ置換（例：`一` → `1`, `十` → `10`）。
    3.  ハイフン、マイナス、全角半角スペース等の不要な記号文字を除外。
    4.  英字はすべて大文字へ統一。
*   **目的**: 「1-1-02」「1年1組2番」「1102」「1 1 02」といった入力を「1102」などの同一キーに統一し、マッチングミスを抑制します。

### 2.2 エンコーディングの自動判別とCSVパース
Excelから出力されたCSVファイルは、環境（OSや保存オプション）によって `UTF-8`（BOMあり/なし）または `Shift-JIS` で保存されます。

*   **関数**: `detectAndReadText(file, callback)`
*   **判別方法**: 
    ファイルのバイナリデータ（`Uint8Array`）を先頭からスキャンし、マルチバイト文字のバイトパターンから簡易的に `UTF-8` 適合性を判定します。適合しない場合は `Shift-JIS` としてデコード（`TextDecoder`）を行います。
*   **制限事項**: 判定ロジックは簡易的なパターンマッチングに基づいているため、特定の極めて短いテキスト等では誤判定が起こる可能性があります。そのため、本システムでは「メモ帳からテキストをコピー＆ペーストする」手順を安全性の高い代替策（デフォルト手順）として提供しています。

### 2.3 ID/PIN の一意生成ロジック
チケットのIDおよびPINコードの生成は、重複を回避しながら指定文字プールからランダム生成します。

*   **関数**: `generateData()`
*   **重複確認**: 
    生成されたID/PINが、これまでの生成履歴（LocalStorage内の `usedIds`, `usedPins`）に存在しないか、`Set` オブジェクトを用いて照合します。
*   **Excelバグ対策**: 
    文字列 `e` や `E` がID等に含まれると、Excelで読み込んだ際に「指数表記（例: 1.23E+04）」に誤変換され、桁落ちや破損を招くことがあります。これを防止するため、文字プール構成段階でデフォルト値として `e` および `E` を明示的に除外しています。

### 2.4 データ結合とルールエンジン (制限・カット処理)
Formsデータ（保護者・一般公開）の紐付け処理を行う主要ロジックです。

*   **関数**: `processData()`
*   **処理フロー**:
    1.  **重複回答の排除**: 
        同一生徒（標準化された年組番号キー）から複数の回答がある場合、マッピングされた「完了時刻」列を比較し、**最新の回答データのみ**を採用します。
    2.  **マスター名簿照合**: 
        マスター名簿は `年組番号 / ID / 許可した土曜券の枚数` の3列で扱います。該当する年組番号キーが存在しない場合、またはIDが空の場合はチケットを発行せず、重要警告としてログおよびUIに出力します。
    3.  **保護者公開日上限カット**: 
        無制限設定が無効な場合、設定された上限値を超える申請数を自動的に上限値まで引き下げ、カットされた人数分を警告として記録します。
    4.  **一般公開日（招待客）上限カット**: 
        マスター名簿から取得した「許可した土曜券の枚数」の上限を超えてFormsから招待客名が申請された場合、**先頭に記述された申請者から優先して採用**し、上限を超える後続の申請者を自動的に除外します。採用された各チケットには同一生徒IDと、PINリストから順番に取得した個別PINを割り当てます。

## 3. 永続化データの構造

LocalStorageには、以下のキーとデータ構成でJSON文字列が保存されます。

*   **Key**: `unified_ticket_generator_data`
*   **構造**:
    ```json
    {
      "settings": {
        "id": {
          "length": 4,
          "useNum": true,
          "useUpper": false,
          "useLower": false,
          "useSym": false,
          "exclude": "eE",
          "excludeFirst": "0"
        },
        "pin": { ... },
        "generateCount": 100
      },
      "history": [
        {
          "batchId": "BATCH_1700000000000",
          "date": "2026/02/08 12:00:00",
          "count": 100,
          "idRules": { ... },
          "pinRules": { ... },
          "data": [
            { "id": "A123", "pin": "9876" },
            ...
          ]
        }
      ],
      "usedIds": ["A123", "B567", ...],
      "usedPins": ["9876", "5432", ...]
    }
    ```

## 4. 保守および改修時の注意点

### 4.1 LocalStorage の容量上限に関する制限
一般的なモダンブラウザにおける LocalStorage の容量上限は約 **5MB** です。
*   生成したチケットの履歴（`usedIds`, `usedPins` および `history`）が累積し数万件規模に達すると、上限エラー（QuotaExceededError）が発生する可能性があります。
*   コード内には、容量過多に対応するための「履歴バックアップ保存(JSON)」および「全履歴リセット」機能が設けられています。
*   また、履歴データが生成後6ヶ月を超えている場合に、ページ読み込み時にリセットを促すダイアログ表示（`checkHistoryExpiry()`）が実装されています。運用方針に応じて、この確認期間を調整してください。

### 4.2 文字プールの枯渇警告
ID・PINの桁数が小さく、使用文字種が少ない状態で大量の生成を要求すると、ランダム生成のループが無限ループ化するか、著しくパフォーマンスが低下します。
*   UI上の変更イベント等にフックされた `checkTheoreticalLimits()` が、理論的な限界パターンの50%を超える要求に対して「赤色警告（生成不可・危険）」、10%を超える要求に対して「オレンジ色警告（注意）」を表示します。
*   文字種のチェックボックスの有無や除外文字の設定を変更する場合は、この計算ロジックが破綻しないよう注意してください。

### 4.3 フォーム設問変更への追従
Formsの設計変更（設問の追加や名称変更）があった場合でも、本システムは「Forms列の紐付け（マッピング）機能」により、コード自体を修正せずに対処できるように設計されています。
*   自動認識をよりスムーズにするためには、マッピング定義（`buildDynamicMapDefinitions` 内の `keys` 配列）に、想定される新しいキーワードを追加してください。
*   結合されたデータは、最終的に `system_import.csv`（縦持ちレコード）および `print_merge.csv`（横持ちレコード）に変換されます。差し込み印刷用データの列ヘッダーの変更は、配付用Wordテンプレート（差し込み印刷設定）の修正を伴うため、慎重に行ってください。
