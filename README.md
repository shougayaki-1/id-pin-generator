# 開成祭チケット発行システム (README.md)

本システムは、学校祭（開成祭）における入場チケット管理を安全かつ効率的に行うために開発された、完全クライアントサイド動作のWebツールです。  
チケットの「空番号（ID/PIN）のランダム生成」と、「Forms回答データとID/PINの紐付け・配付制限枠に基づく自動足切り」の2つの主要機能を1ファイルに集約しています。

---

## 1. 設計思想とアーキテクチャ

本システムは、学校の運用環境（毎年PC環境や担当教員が変わり、インターネット制限や個人情報保護の規則が厳しい状況）を考慮し、以下の設計思想に基づいて実装されています。

* **ゼロ・ディペンデンシー (Zero-Dependency)**
  * jQueryやReact、Vue.jsなどの外部ライブラリ、Webフォント、CDNを一切使用せず、標準のVanilla JSおよびCSSのみで構築されています。
* **プライバシー・バイ・デザイン (Privacy by Design)**
  * アップロードされた生徒情報やFormsの回答データは、ブラウザ内のメモリ上（Javascriptの配列等）でのみ処理されます。外部サーバーへの送信処理は一切含まれておらず、ネットワークの切断された完全なオフライン環境でも動作します。
* **高ポータビリティ (Single-file Portability)**
  * HTML / CSS / JS、および解説用マニュアルが、すべて単一の `.html` ファイルにパッケージングされています。この1ファイルをUSB等で配布するだけで動作環境が構築できます。

## 2. 技術仕様とデータフロー

### 2-1. ID/PIN生成エンジン (Tab 1)
* **永続状態の管理**:
  過去に生成したIDやPINは、ブラウザの `localStorage` にシリアライズして保存されます。これにより、同一ブラウザで複数回に分けて生成しても、IDの二重発行を防ぐことができます。
* **重複回避ロジック**:
  ランダム文字列の生成時に、`usedIdsSet` および `usedPinsSet` を用いて重複照合を行います。生成確率の衝突が起きた場合は、最大2,000回まで試行ループを回す無限ループ防止機構（セーフティネット）を実装しています。
* **指数表記化バグの回避**:
  ExcelでID/PINリストを開いた際、数字と特定のアルファベットの組み合わせ（例: `e` や `E`）が指数表記（「`1.00E+04`」など）として自動認識・破壊されてしまうのを防ぐため、文字プールからあらかじめ `e` / `E` を取り除いた除外ロジックを採用しています。

### 2-2. 結合・クレンジング処理 (Tab 2)
本機能は、以下の4つのCSVを読み込み、メモリ上でリレーショナル処理します。

```
[① 制限マスター]  ──(照合)──┐
[② 保護者Forms]  ──(名寄せ)──┼──► [メモリ上での結合処理] ──► [CSV出力 (マスタ用 / 印刷用)]
[③ 招待者Forms]  ──(名寄せ)──┤
[④ 空ID/PIN]     ──────(割当)──┘
```

* **最新回答の自動採用（名寄せ）**:
  同一生徒がFormsを誤って複数回送信した場合に備え、「年組番号」を主キー（MapのKey）とし、完了時刻（タイムスタンプ）を比較することで、**常に最も日時の新しい回答のみをメモリ上に保持する（名寄せ）**仕様となっています。
* **枠制限（足切り）ロジック**:
  一般公開日の招待者数が「割り当てマスター」で指定された制限枠数（Quota）を超過している場合、先頭から順に枠数分のみをチケット割当とし、超過した招待客は自動的にカットされます。
* **警告・是正データの構造化**:
  足切りが発生した不整合データは、オブジェクト（`flaggedIssues`）にプールされ、画面上にアコーディオンUIとして構造化表示されるほか、Wordでの「個別お詫び通知書」の差し込み印刷ソース等に使えるよう、個別にCSV形式（`warnings_list_adjust.csv`）でエクスポートする機能を設けています。
* **入力枠（列数）の動的スキャン**:
  保護者の来場数は原則無制限ですが、CSVの構造を崩さないために、アップロードされたCSVのヘッダーから「入場者●（または招待者●）」という名称パターンを検出し、その年の**最大列数を自動的に検出・UIマッピングへ反映**します。
* **フリガナ（カタカナ読み）エンジン**:
  英単語の読み間違い（例: ログインPIN入力時の「C」と「シー」「シ」の誤認など）を防ぐため、1文字ずつカタカナ音にパースする辞書マッピング処理を行っています。

## 3. 保守・拡張へのガイド

引き継ぎ後の仕様変更時、ソースコード内のどのエリアをカスタマイズすべきかをまとめています。

### 3-1. PINのカタカナの読み方を修正したい場合
コード内の `phoneticMap` オブジェクトを変更してください。例えば「V」を「ブイ」から「ヴィー」にするなどのチューニングが可能です。
```javascript
const phoneticMap = { 'A':'エー', 'B':'ビー', 'C':'シー', ... };
```

### 3-2. 自動読み込みのキー（キーワード）を追加・変更したい場合
Formsの設問名が変更され、列マッピングの「自動推測」が働きづらくなった場合は、`buildDynamicMapDefinitions` 関数や `buildMappingUI` 関数が評価する検索用キーワード配列（`keys`）にキーワードを追加してください。
```javascript
// 例：年組番号を検索するためのキーワード群
{ id: prefix + '_Class', label: '年組番号(必須)', keys: ['年組番号', 'クラス', '学級', '新しい設問名'] }
```

### 3-3. 出力CSVが他の発券システムで文字化けする場合
本システムは、Microsoft Excelでの安全なダブルクリック閲覧を想定して、出力CSVの先頭にBOM（Byte Order Mark）を付与しています。
```javascript
const bom = new Uint8Array([0xEF, 0xBB, 0xBF]); // UTF-8 BOM
```
もし他のWebサーバーやLinuxベースのデータベースシステムに流し込む際、BOMが原因でエラーが出る場合は、出力時の `triggerDl` 関数、または `exportBatchToCSV` 内の `new Blob([bom, ...])` から `bom` を除外してプレーンなUTF-8として出力させてください。
